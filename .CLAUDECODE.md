# .CLAUDECODE.md — Code Conventions & Standards

> Read this file before writing any code for the Nail Salon project.
> These are the non-negotiable conventions Claude must follow throughout this codebase.

---

## General Principles

- Write TypeScript everywhere. No `any`. No implicit `any`.
- Prefer explicit types over inferred types for function signatures.
- No `console.log` left in committed code. Use a logger in the API.
- Do not add comments unless the logic is genuinely non-obvious.
- Do not add docstrings, JSDoc, or type annotations to code you did not write.
- Do not create helpers or utilities for one-time use. Keep abstractions minimal.
- Do not add error handling for scenarios that cannot happen.
- Validate at system boundaries only (user input, external APIs, webhook payloads).

## TypeScript Rules

```ts
// GOOD: explicit return types on exported functions
export function formatPrice(cents: number): string { ... }

// BAD: implicit any
const handler = (req, res) => { ... }

// GOOD: typed Express handlers
import { Request, Response } from 'express'
const handler = (req: Request, res: Response): void => { ... }

// GOOD: Zod for runtime validation at boundaries
const schema = z.object({ email: z.string().email() })
const parsed = schema.parse(req.body)

// BAD: casting without validation
const body = req.body as BookingInput
```

## Component Conventions (React / Next.js)

- All components are **function components**. No class components.
- Use **named exports** for components. No default exports for components.
- Co-locate component-specific types in the same file unless shared across multiple files.
- Use `cn()` (from `lib/utils.ts`) for conditional class merging with Tailwind.
- Use Shadcn/ui base components as the foundation. Extend them, do not reinvent them.
- Do not use inline styles. Use Tailwind classes or CSS variables.
- Do not hardcode colors as hex values. Always use CSS variables defined in `globals.css`.

```tsx
// GOOD
export function ServiceCard({ name, price }: ServiceCardProps) {
  return (
    <div className={cn('rounded-lg border bg-card p-4', className)}>
      ...
    </div>
  )
}

// BAD
export default function serviceCard(props) {
  return <div style={{ color: '#f43f5e' }}>...</div>
}
```

## State Management

- **Zustand** for global client state (booking flow, UI preferences).
- **TanStack Query** for all server data (fetching, caching, mutations).
- Do not use `useState` for data that belongs in TanStack Query.
- Do not use TanStack Query for UI-only local state.

```ts
// Zustand store pattern
interface BookingStore {
  selectedService: Service | null
  selectedStaff: Staff | null
  selectedDate: Date | null
  setService: (s: Service) => void
}

export const useBookingStore = create<BookingStore>((set) => ({
  selectedService: null,
  selectedStaff: null,
  selectedDate: null,
  setService: (s) => set({ selectedService: s }),
}))
```

## API & Data Fetching

- All API calls from the frontend go through `apps/web/lib/api.ts` (typed fetch wrappers).
- Never call `fetch()` directly in a component. Always use TanStack Query + the `api.ts` helpers.
- API responses must match a Zod schema before being used.

```ts
// lib/api.ts pattern
export async function getServices(): Promise<Service[]> {
  const res = await fetch(`${process.env.NEXT_PUBLIC_API_URL}/services`)
  if (!res.ok) throw new Error('Failed to fetch services')
  const data = await res.json()
  return serviceArraySchema.parse(data)
}
```

## Express API Conventions

- Routes → thin. Only call controller function.
- Controllers → validate input (Zod), call service, return response.
- Services → business logic, DB queries via Prisma.
- Never write raw SQL strings. Always use Prisma's query API.

```ts
// Route (thin)
router.post('/bookings', authMiddleware, createBookingController)

// Controller
export async function createBookingController(req: Request, res: Response) {
  const input = createBookingSchema.parse(req.body)
  const booking = await createBookingService(input, req.user.id)
  res.status(201).json(booking)
}

// Service
export async function createBookingService(input: CreateBookingInput, userId: string) {
  return prisma.appointment.create({
    data: { ...input, userId, status: 'PENDING' }
  })
}
```

## Error Handling

- Express: use a centralized error handler middleware. Controllers throw; middleware catches.
- Next.js: use `error.tsx` boundary files per route segment for UI errors.
- Never swallow errors silently.
- Never expose internal error details (stack traces, DB errors) to the client in production.

```ts
// Express centralized error handler (in server.ts)
app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
  console.error(err)
  res.status(500).json({ error: 'Internal server error' })
})
```

## Authentication

- Use Clerk (preferred) or Auth.js for client-facing auth.
- JWT issued by the Express API for session management (stored in HttpOnly cookies).
- Admin routes: protected by `requireRole('ADMIN')` middleware.
- Client routes: protected by `requireRole('CLIENT')` middleware.
- Never store tokens in `localStorage`.

## Import Order (enforced by ESLint)

1. Node built-ins
2. External packages
3. Internal aliases (`@/components/...`, `@/lib/...`)
4. Relative imports (`./`, `../`)
5. Type imports (always last)

## Git Commit Style

- Format: `type(scope): short description`
- Types: `feat`, `fix`, `chore`, `refactor`, `style`, `test`, `docs`
- Keep subject under 72 characters.
- No period at the end.

```
feat(booking): add service selection step to wizard
fix(auth): redirect unauthenticated users to sign-in page
chore(deps): add framer-motion
```

## What Claude Must Never Do

- Never commit `.env`
- Never use `any` in TypeScript
- Never write raw SQL strings
- Never store auth tokens in `localStorage`
- Never hardcode hex colors inline
- Never install packages without asking permission
- Never create files without asking permission
- Never modify existing files without asking permission
- Never skip Zod validation at API boundaries
- Never add speculative features, helpers, or abstractions
